Few-Shot Detection

Training with Only 1.0‰ Samples: Malicious Traffic Detection via Cross-Modality Feature Fusion

极低标注成本（仅千分之一样本）实现高精度恶意流量检测；跨模态融合Packet/Flow/Host三级特征；拓扑驱动对比预训练消除不同网络的偏差
ACM CCS 2025 Chuanpu Fu et al. (清华大学 & Purdue大学) 系统名: tFusion github.com/fuchuanpu/TFusion

1. 研究动机与问题定义

1.1 核心问题：标注成本阻碍大规模部署

基于机器学习的恶意流量检测已经发展了十余年，在学术实验和特定部署场景中展现出了超越传统规则检测的能力。然而，每当要将检测系统部署到一个全新的网络环境时，都会面临一个根本性的瓶颈：

标注饥渴问题：要在新网络中训练一个有效的ML检测模型，安全专家需要捕获并标注数百万条良性和攻击流量。这个过程涉及深度协议知识、攻击行为判断、特定场景理解，极其耗时且昂贵。

全球恶意流量检测市场规模超过 36.4 亿美元，但对绝大多数网络来说，承担大规模流量标注的人力成本是不现实的。这导致大量网络无法获得有效的ML驱动安全防护。

1.2 "1.0‰ 样本" 意味着什么 —— 具体示例

1.0‰（千分之一）= 0.1% 的标注率

具体场景：一个企业网络在 5 天内产生了约 91,000 条流。传统方案需要标注至少 10,000-50,000 条流（10-50%）才能训练有效模型。tFusion 仅需标注91 条流（= 91,000 × 1.0‰）即可达到 99.82% 的检测准确率。

更直观的对比：
- 传统方案：安全团队需要数天到数周标注数万条流
- tFusion：安全团队可以在 几十分钟内完成 91 条流的标注

另一个极端场景：从 50 个样本（~0.5‰ 量级）学习时，tFusion 仍然保持 >94.92% 的精度。

1.3 为什么现有方法无法解决这个问题

方法一：从其他网络引入数据

不同网络的流量特征分布差异极大（拓扑、用户群、应用组合、时段模式都不同），来自网络 A 的标注数据在网络上 B 可能完全失效。Figure 1(a) 的 t-SNE 可视化清楚展示了这一点——不同数据集的流特征分布几乎没有重叠。
方法二：数据增强

现有的流量数据增强方法（如对 Tor 流量或 TLS 流量的增强）只能生成特定类型流量，依赖先验知识。恶意流量检测必须处理各类协议和应用，单一类型的数据增强无法覆盖这种多样性（实验证明数据增强方法的性能显著低于 tFusion）。
tFusion 的核心洞察：现有方法将流量视为单模态 (unimodal) 数据——要么只看 per-packet 特征，要么只看 per-flow 特征，要么只看 per-host 特征。这种单一粒度的视角导致了信息的大量丢失，迫使模型依赖大量标注数据来弥补信息不足。

解决方案：将网络流量视为多模态数据——从 packet（细粒度序列特征）、flow（中粒度统计特征）、host（粗粒度交互特征）三个视角同时提取特征，通过跨模态融合让有限标注数据发挥出丰富的表征能力。

2.1 三模态视角：Packet / Flow / Host

tFusion 的核心创新在于将网络流量视为三种不同粒度模态的组合：

模态粒度特征类型提取内容维度示例
Packet 模态 细粒度 / 序列特征 每包的长度 + 包间到达时间 时间感知的位置编码 + 长度嵌入 序列长度 N（每流包数）
Flow 模态 中粒度 / 非序列统计特征 每流的聚合统计量 包数、字节数、FCT、最大/最小/平均包长 6 维特征向量
Host 模态 粗粒度 / 非序列交互特征 同一主机的多流交互模式 同源/同目的的流数、字节数、包数统计 12 维特征向量
为什么这三个粒度互补？

- Packet 序列捕获攻击的时序行为模式（如脉冲式 DoS、周期性 C2 心跳），但对网络拓扑和用户行为敏感。
- Flow 统计提供每条流的聚合视图（如一个流的总数据量、持续时间），稳定但丢失了时序细节。
- Host 交互提供主机级别的通信模式（如在短时间内某个主机与多少个不同目的通信——是端口扫描还是正常浏览），视角最粗但抗单流噪声最强。

三者的跨模态融合让模型同时拥有"显微镜"（packet）和"望远镜"（host）的视角，即使数据很少也能构建稀疏但有效的决策边界。

3. 模块一：Time-Aware Packet Sequence Embedding

3.1 为什么需要时间感知编码

传统的 Transformer 位置编码（如 BERT 的正弦位置编码）仅编码了 token 的顺序（第1个、第2个、...），但完全忽略了包之间的时间间隔。

在恶意流量检测中，时间间隔是关键信号：

  • 连续小间隔（<10ms）的包 → 可能是一次突发攻击
  • 固定大间隔（如每 60 秒）的包 → 可能是 C2 信标
  • 随机间隔 → 正常用户交互

3.2 时间感知位置编码算法

Step 1: 包长度嵌入
  对每个包 j（属于流 i）：
    偏移长度 = 协议标识符 × 最大包长 + min(实际长度, 最大包长) - 1
    → 通过可学习的 Embedding 层映射为浮点向量

  目的：协议标识符（TCP=6, UDP=17）作为偏移让不同协议的包在嵌入空间中有不同的基础位置。

Step 2: 时间感知的位置编码
  给定包序列的到达时间戳序列 ®t_i = [t_1, t_2, ..., t_N]：

  u_j = 0                         如果 j = 1（流中的第一个包）
  u_j = 2 × j + 1                 如果 t_j - t_{j-1} > T（大间隔，标记"显著间隔"）
  u_j = 2 × j                     否则（正常间隔）

  其中 T 是时间阈值参数。

  然后对 u_j 应用正弦函数：
    U_{j, 2p}   = sin(p × e^{-j × ln(T)/H})
    U_{j, 2p+1} = cos(p × e^{-j × ln(T)/H})

  关键设计：指数衰减 e^{-j × ln(T)/H} 让时间信号的频率随位置变化，
  早期位置 → 高频（敏感于小时间差）
  后期位置 → 低频（关注大时间模式）

Step 3: 组合
  S_i_j = T_i_j + E_i_j
  → 包 j 的最终嵌入 = 时间位置嵌入 + 长度嵌入

3.3 序列特征提取 (Self-Attention)

对组合后的包序列 S(i) 应用小型自注意力模型（比语言模型的注意力层更小，以减轻过拟合）：

Q = Linear(S; W_Q, b_Q)    # Query 矩阵
K = Linear(S; W_K, b_K)    # Key 矩阵
V = Linear(S; W_V, b_V)    # Value 矩阵

Attention = softmax(Q × K^T / √H) × V

输出 = GELU(Linear(Attention + S; W_1, b_1))
最终 = Softmax(Linear(MLP_output; W_2, b_2)) + Attention  # 残差连接

取输出的第一维向量 ®p_i = 该流的细粒度序列特征表示
注意力可视化揭示的洞察（论文 Figure 3 & 4）：
- 良性流量的注意力矩阵：分布相对均匀，无明显的自关注或周期性模式
- 恶意流量的注意力矩阵：呈现明显的规则周期性模式（如脉冲 DoS 攻击的周期性突发、Telnet 注入攻击的特定区域强化注意力）
- 恶意流量的平均注意力值显著高于良性流量——因为攻击行为往往比正常行为更"规则"和"可预测"
- 对角线上无显著自关注（不同于 NLP 中的 token 自关注偏好），说明包之间的跨位置关系更重要

4. 模块二：Crossmodal Attention Feature Fusion

4.1 Flow 统计特征

对每条流 F_i，从 per-packet 特征矩阵提取 6 维 flow-level 统计量（经对数变换提高数值稳定性）：

®f_i = Log([
    N_i,                    # (1) 包数
    t_Ni - t_1,             # (2) 流完成时间 (FCT)
    sum(l_1..l_Ni),         # (3) 总字节数
    min(l_1..l_Ni),         # (4) 最小包长
    max(l_1..l_Ni),         # (5) 最大包长
    mean(l_1..l_Ni)         # (6) 平均包长
] + 1)

4.2 Host 特征

对时间窗口 L 内到达的所有流，提取 12 维 host-level 交互特征：

源/目的主机的发送和接收模式：
  send(h) = [
    |fsrc(h; F)|,                  # 以 h 为源的流数
    sum_{F_j in fsrc(h)} N_j,      # 以 h 为源的总包数
    sum_{F_j in fsrc(h)} bytes_j,  # 以 h 为源的总字节数
    ...                             # 以及均值/方差等派生统计量
  ]
  receive(h) = [类似地对 dst(h) 计算]

6 个源侧特征 + 6 个目的侧特征 = 12 维

4.3 Cross-Attention 融合机制

这是 tFusion 与 MM4flow 在融合设计上的核心差异：

tFusion 的 Cross-Attention 设计：

  Query  = Host Features  (粗粒度，指导注意力)
  Key    = Flow Features  (中粒度，提供匹配目标)
  Value  = Packet Features (细粒度，被融合的信息载体)

计算过程：
  1. Q = Linear(Host_Features)      # (batch, H) → (batch, H)
  2. K = Linear(Flow_Features)      # (batch, H) → (batch, H)
  3. V = Packet_Features             # 直接使用，不经线性变换

  4. Attention = softmax(Q × K^T / √H)  # (batch, batch) — 样本间注意力
  5. Fused = Attention × V              # (batch, H)

  6. Final_Feature = Fused + Flow_Features  # 残差连接

设计哲学：
  Host 特征（如"这个主机在短时间内连接了多少个不同目的"）
  作为 Query，去"查询"哪些 Flow 特征（如"这个流的包长分布"）
  是相关的，最终从 Packet 序列特征中提取最相关的信息。

  这种设计保证了：
  - 粗粒度特征（Host）提供"全局上下文"
  - 中粒度特征（Flow）提供"匹配基准"
  - 细粒度特征（Packet）提供"信息内容"

4.4 与 MM4flow 的 Cross-Attention 对比

维度tFusion (CCS 2025)MM4flow (CCS 2025)

融合粒度 多粒度（细/中/粗） 双粒度（字节级/包级）

关键差异总结：tFusion 的融合更接近"结构化检索"——用粗粒度的 host 信息去"查询"哪些 packet 序列细节是关键的；而 MM4flow 的融合更接近"无约束的信息交换"——所有 token 之间自由互相关注。tFusion 的交叉注意力有明确的语义角色分配（Q=Host, K=Flow, V=Packet），且在预训练阶段就开始训练，这使其在极低标注场景下更具优势。

5. 模块三：拓扑驱动对比学习预训练

5.1 为什么需要特殊的预训练策略

标准对比学习（如 SimCLR）依赖数据增强来构造正样本对。但流量数据中：

  • 如何定义"同一个流的不同视角"？（不能像图像一样裁剪/翻转）
  • 如何避免学到网络拓扑的偏差而非攻击行为？（不同网络的 IP 分布完全不同）

5.2 拓扑驱动的对比学习方案

核心思想：利用流量生成地址（哪个 IP/主机生成了这条流）作为天然的对比监督信号。同一个地址生成的流量视为正样本对（应被拉近），不同地址生成的流量视为负样本对（应被推远）。

这一设计优雅地绕过了标注需求——地址信息在数据采集中天然可得，无需人工标注。同时，因为预训练目标是"无论什么地址，聚类同地址流量"，而非"记住特定地址的特征"，它能消除对不同网络拓扑的过拟合。 
拓扑驱动对比损失的训练过程：

给定 batch = {F_1, F_2, ..., F_B} 个流：
  1. 通过时间感知嵌入 + crossmodal attention 提取每个流的特征 z_i
  2. 对于每个流 i：
      正样本: 同一 batch 中与 i 共享源地址的其他流
      负样本: 同一 batch 中与 i 不同源地址的流

  3. 对比损失：
     L_i = -log(
         Σ_{j∈Positive(i)} exp(sim(z_i, z_j) / τ) /
         Σ_{k≠i} exp(sim(z_i, z_k) / τ)
     )

  sim(a,b) = cosine similarity
  τ = temperature parameter

总损失：L_total = (1/B) × Σ_i L_i

在整个预训练过程中：
  - 模型逐渐学会将同一地址生成的流量聚集在特征空间中
  - 但不会记忆特定地址的绝对位置
  - 这相当于在学"什么样的流量模式倾向于来自同一主机"
    → 对于检测："一个主机突然产生与平时完全不同模式的流量" → 可能是被攻陷
与 CLIP 的类比：
CLIP 使用 4 亿对 (图像, 文本) 配对作为正样本——利用 alt-text 作为免费监督信号。
tFusion 使用源地址作为免费监督信号——利用"同一地址 = 正样本对"这个网络拓扑中的天然事实。

两者的共同智慧：在预训练阶段不需要类别标签，只需要一个合理定义的"正/负样本对"规则，就能学到有用的表征。

5.3 预训练数据

使用大规模无标注的互联网流量数据集进行预训练。关键优势：预训练数据不需要来自目标部署网络——拓扑驱动对比学习的目标正是消除对特定拓扑的依赖。预训练好的模型可以复用到多个不同网络的部署场景，仅需各网络自身极少量的标注样本进行微调。

6. 模块四：Lightweight ML Detection (AutoML)

6.1 为什么用 AutoML 而非端到端训练分类器

tFusion 的预训练模型产出的融合特征向量（Fused Feature Vector），已经包含了丰富的跨模态表征信息。此时训练一个分类器不需要复杂的深度网络——传统的轻量级 ML 模型足以胜任。

使用 AutoML 框架自动选择和调优最佳模型：

场景可选模型场景特点
有监督
（有攻击样本标注）		 Random Forest, XGBoost
SVM, Decision Tree
（nPrintML, FlowLens, Taurus, NetBeacon 使用的模型）		 需要标注部分攻击流量；可精确分类攻击类型
无监督
（仅良性样本）		 K-Means, AutoEncoder
Isolation Forest, Graph Learning
（Whisper, Kitsune, HorusEye, HyperVision 使用的模型）		 更贴近实际：攻击样本通常未知且多样；检测异常偏离

AutoML 自动遍历候选模型和超参数组合，选择在验证集上表现最佳的模型。这避免了针对特定场景手工选模型的负担，实现了"部署即自动适配"。

关键洞察：tFusion 的检测模块本身不引入新的深度学习复杂度。核心性能增益来自预训练的 crossmodal attention 提取的融合特征，而非分类器的复杂性。34ms 的端到端检测延迟证明了这种"重特征提取 + 轻分类器"设计的工程可行性。

标注需求 中（10-30% 标注达全量水平） 极低（1.0‰ 标注即达 99.82%） tFusion 在标注效率上更激进
模型规模 174M（双 BERT） 较小（自注意力 + AutoML 轻量分类器） tFusion 更注重部署效率
推理延迟 未报告 34ms tFusion 更注重实际运维指标
融合时机 仅微调阶段 预训练阶段即开始 tFusion 的融合更早期、更深度
时间信息 排除（认为不稳定） 核心编码（时间感知位置编码） 对时间信息的相反立场
payload 使用 核心模态（Byte） 不使用（易被攻击者操纵） 安全性 vs 信息量的权衡
两种范式的本质：
- MM4flow 走的是 "大预训练 + 通用表征"路线——用海量数据训练出能适配多种分类任务的通用模型，类似 NLP 中的 BERT。
- tFusion 走的是 "轻量预训练 + 高效适应"路线——用对比学习在无标注数据上预训练，然后在新网络中以极少标注快速适配，类似 few-shot learning 的思路。

两者的互补性意味着未来可以结合：用 tFusion 的 crossmodal attention 设计（Q=Host, K=Flow, V=Packet）替换 MM4flow 的 concat-query cross-attention；在 MM4flow 的 77.6TB 数据上做拓扑驱动的对比预训练，有可能同时获得"通用表征"和"极低标注适应性"。

9. 批判性分析

9.1 潜在问题一：不依赖 Payload 是优势还是局限？

tFusion 明确不使用 raw payload bytes，理由是"攻击者可以轻易操纵原始字节进行逃逸攻击"。这是一个合理的防御性设计选择，但也带来了信息丢失：

- 对于 TLS 握手明文阶段的恶意特征（如异常的密码套件组合、自签名证书），payload 中包含的关键信号被完全忽略。
- 对于 应用层攻击 payload（如 SQL 注入、XSS 在 HTTP 请求体中），单纯的包长和流统计无法检测。
- 这种设计使得 tFusion 更适合检测 行为层面的异常（如扫描、DDoS、C2 通信模式），但对于 内容层面的攻击（payload 注入类）能力有限。

但这可能是一个可以接受的权衡：在极低标注限制下，paylaod 特征的复杂性可能超过有限标注数据的建模能力。tFusion 选择的行为特征恰好是"用少量样本即可刻画"的。

9.2 潜在问题二：Host 特征的窗口依赖性

Host 特征基于时间窗口 L 内的流聚合。窗口大小的选择直接影响特征质量和检测性能：

- 窗口太小 → host 交互模式不完整（可能只捕获了交互的一半）
- 窗口太大 → 历史信息稀释当前行为的异常信号
- 论文未详细讨论 L 的选择标准和敏感性分析

此外，在高速网络（100Gbps+）中，时间窗口内可能同时有数万至数十万条流，host 特征的实时计算可能成为瓶颈。

9.3 潜在问题三：拓扑驱动对比学习的假设

其核心假设是"同一地址生成的流量在特征空间中应被聚类"。这个假设在多数场景成立，但在某些攻击场景下可能适得其反：

- IP 欺骗攻击：攻击者伪造源 IP，导致模型错误地将攻击流量与良性主机关联。
- NAT/代理环境：大量用户共享同一公网 IP，模型会试图聚类完全不同的用户行为，这可能产生噪声表征。
- 多宿主主机：同一物理主机可能使用多个 IP（如双栈 IPv4/IPv6），模型无法自然地关联这些流量。

这些限制论文在第 6 节 Limitations 中有部分讨论，但缺少定量实验来衡量这些场景下的性能退化程度。

9.4 潜在问题四：在三模态之间如何防止"模态偏差"？

MM4flow 引用多模态偏差作为"不做 joint pre-training"的理由。tFusion 在预训练阶段就进行了三模态融合，但没有明确讨论如何防止某一模态（如容易学习的 flow 统计特征）支配其他模态（如需要更多数据学习的 packet 序列特征）。

Cross-attention 中 Q=Host, K=Flow, V=Packet 的设计在一定程度上通过固定语义角色限制了模态竞争（每个模态有明确的"职责"），但并未主动检测或调节模态间的梯度不平衡。

9.5 潜在问题五：浅层 Baseline 的对比充分性

tFusion 对比了 14 种 SOTA 方法（包括 nPrintML、FlowLens、Whisper、Kitsune、HyperVision 等），这些方法本身已经是精心设计的安全系统。但与 Sugar (SIGCOMM 2025) 的批评呼应：一个简单的 Random Forest + 精心手工设计的特征是否也能在 1.0‰ 样本下达到有竞争力的性能？如果 simple baseline 表现尚可，那么 crossmodal attention 的相对增益就需要更严格的核算。

不过，tFusion 的 AutoML 模型选择已经包含 RF/XGBoost/SVM 等传统模型——它不是在"DL vs 浅层"之间比较，而是证明了"tFusion 提取的融合特征 + RF"比"原始特征 + 各种模型"更好。

9.6 积极评价

tFusion 的设计中有几个值得强调的优点：

1. 实际部署验证：在一个有 140+ 活跃用户的生产网络中进行 5 天部署，这是论文结果中最具说服力的部分。与纯粹的离线实验相比，真实网络环境中的不可控因素（网络波动、用户行为变化、新流量模式）让 99.82% 这个数字更有分量。

2. 检测延迟报告：34ms 的端到端延迟明确量化了部署可行性。相比之下，MM4flow 未报告推理延迟，Sugar 论文专门批评了 DL 模型在高速网络中的推理瓶颈。

3. 无监督检测支持：支持仅用良性样本训练无监督检测模型——这是实际部署中最常见的场景（攻击样本难以获取和标注）。

4. 开源代码和数据：https://github.com/fuchuanpu/TFusion 公开了完整实现，有利于社区复现和改进。

10.2 与 broader 趋势的关联

  • 与 SoK (IEEE S&P 2025) 的关联：tFusion 不使用 payload bytes，避免了 SoK 指出的"加密 payload 中可学习模式存在争议"问题。其行为特征（包长、时间、流统计）在 TLS 1.3 和加密隧道下仍然有效。
  • 与 Sugar (SIGCOMM 2025) 的关联：tFusion 使用 AutoML 选择分类器（包含 RF/XGBoost），而非纯 DL 端到端分类，一定程度上回应了 Sugar 对"浅层模型可以很强大"的论证。
  • 与 MM4flow (CCS 2025) 的关联：如上节所述，两者代表了"多模态流量分析"的两种互补范式，融合两者可能是未来的方向。

10.3 实际应用前景

tFusion 最可能的应用场景：
1. 中小企业网络安全：没有专业安全团队，无法承受大规模标注。部署 tFusion，仅标注数十条流即可获得有效检测。
2. 多网络部署：ISP 或云服务商需要在数千个客户网络中部署检测。用互联网数据做一次预训练，然后在每个客户网络中用极少标注微调。
3. 快速应急响应：新攻击出现时，可以在几分钟内用少量捕获的攻击样本训练检测模型，实现近乎实时的响应。

这三个场景的共同特点是标注资源极度稀缺——而这正是 tFusion 设计的核心假设。

术语定义
