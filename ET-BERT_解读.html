ET-BERT: A Contextualized Datagram Representation with Pre-training Transformers for Encrypted Traffic Classification

第一个将 BERT 预训练范式系统引入加密流量分类的工作；在 7 个数据集 5 类下游任务上大幅超越 SOTA
📄 WWW 2022 👤 Xinjie Lin et al. (中科院信工所) ⭐ 引用 380+ 🔗 github.com/linwhitehat/ET-BERT

1. 研究动机与问题背景

1.1 核心问题：加密流量分类为什么难？

网络流量加密（TLS 1.3、QUIC 等）的普及使得传统的流量分类方法逐步失效。核心矛盾在于：加密保护了隐私，也掩盖了分类所需的特征。具体而言，三条技术路线各有根本性局限：

路线一：DPI / 指纹匹配
依赖明文字段（TLS 握手 SNI、证书 CN、HTTP Host）。TLS 1.3 加密了证书和 SNI（ESNI/ECH），QUIC 从握手阶段即加密。可用明文信息日趋稀疏和被混淆。
路线二：统计特征 + 传统 ML
依赖专家手工设计特征（包长序列、到达间隔、流统计量）。特征设计耗时且难以泛化——新协议、新应用需要重新设计。同一应用在不同网络环境下的统计分布可能剧变。
路线三：有监督深度学习
如 DeepPacket、FS-Net 等从原始流量自动学习特征。但严重依赖大规模标注数据。加密流量的标注成本极高（需要深度协议知识），且训练后的模型容易过拟合特定场景的标注偏差。

1.2 ET-BERT 的核心洞察

受 NLP 领域 BERT 的启发，ET-BERT 提出了一条不同的路径：

核心思想：利用大规模无标注开放域流量数据，通过自监督预训练学习通用的流量数据报表示（datagram representation），然后在下游有标注任务上微调。

类比：在 NLP 中，BERT 从巨量无标注文本学习语言的通用表示；在流量分析中，ET-BERT 从巨量无标注流量数据学习网络通信的通用表示。

这一范式的优势在于：预训练阶段完全脱离标注依赖，下游微调只需要少量标注数据即可达到优异性能（实验证明仅用 10% 数据即可超越全量数据训练的监督方法）。

2. 核心方法

2.1 总体架构

ET-BERT 采用两阶段范式：预训练（无监督/自监督）→ 微调（有监督下游任务）。骨干网络为标准 BERT-base 架构。

2.2 Datagram2Token：从原始流量到 Token 序列

这是 ET-BERT 最关键的工程创新——如何将非结构化的网络数据报转化为 Transformer 可消费的离散 Token 序列。整个过程分三步：

Step 1: BURST Generator

BURST（突发）定义为：同一方向、连续发送的数据包序列。这个概念的灵感来源于 HTML 文档的 DOM 树结构——如同 HTML 通过嵌套标签形成层次结构，网络流量通过方向变化形成自然的"段落"边界。

具体示例：
假设一个 TCP 流的包序列为（C=客户端，S=服务器，箭头表示方向）：
C→S, C→S, S→C, S→C, S→C, C→S, S→C

则 BURST 切分为：
BURST₁ = [C→S, C→S]   （客户端连续发两个包）
BURST₂ = [S→C, S→C, S→C]   （服务器连续回三个包）
BURST₃ = [C→S]   （客户端又发一个包）
BURST₄ = [S→C]   （服务器回一个包）
为什么选用 BURST 而非单包或整个流？
(1) 单包粒度信息太少，尤其对于仅含 ACK 的空包；(2) 全流粒度太长（一条流可能有数千包），超过 Transformer 512 token 上限；(3) BURST 粒度是自然的信息边界——同一方向的连续包通常属于同一协议语义单元（如 HTTP 请求的多个 TCP 分段），方向变化标志协议交互的顺序转折。

Step 2: BURST2Token

将 BURST 内的原始字节转换为 token 序列，采用 Bigram + BPE 双重编码：

Bigram 编码：每相邻两个字节合并为一个 16-bit 整数（0~65535），用 4 位十六进制表示。

原始字节序列（十六进制）：16 03 01 00 F7 03 03 ...
按 Bigram 切分：[16 03] [01 00] [F7 03] [03 ...]
Token 序列：1603, 0100, F703, 03??, ...

选择 Bigram 的理由：单字节（1-gram）词表仅 256，信息密度太低，序列过长；Tri-gram 词表达 16M，过于稀疏。Bigram 词表 65536 在信息密度和稀疏性之间取得平衡。

在 Bigram 基础上，再使用 BPE（Byte Pair Encoding）进一步压缩高频字节对组合，减少序列长度并捕获更高层级的协议模式。最终词表最大 65,536 + 特殊 token（[CLS], [SEP], [PAD], [MASK]）。

Step 3: Token2Embedding

每个 token 的嵌入由三部分组成：

Embedding(token_i) = TokenEmb(token_i) + PositionEmb(i) + SegmentEmb(seg_i)

其中：
- TokenEmb:   可学习的词嵌入矩阵 W ∈ R^(65536 × 768)
- PositionEmb: 标准正弦位置编码，捕获 token 在 BURST 内的顺序
- SegmentEmb:  区分 token 属于前半段（Segment A）还是后半段（Segment B）的 BURST

Segment Embedding 的设计原因：每个 BURST 被等分成两半（sub-BURST A / B），中间插入 [SEP]。Segment Embedding 让模型知道 token 属于哪个半段，这是预训练任务 SBP 所必需的。

2.3 双预训练任务

ET-BERT 设计了两个互补的预训练任务：

Task 1: Masked BURST Model (MBM)

对应 BERT 的 MLM（Masked Language Model），目标是在 BURST 内部学习字节间的依赖关系。

具体操作：
1. 随机选择 15% 的 token 用于掩码
2. 其中 80% 替换为 [MASK]
3. 其中 10% 替换为随机 token（防止模型只学会检测 [MASK] 本身）
4. 其中 10% 保持不变（迫使上下文推理而非仅依赖局部）

损失函数：负对数似然
L_MBM = -(1/|M|) * Σ log P(token_m | context)
仅对掩码位置计算损失，其余位置忽略。
直觉理解：一个 TLS ClientHello 消息的字节序列中，某个位置的加密算法套件（Cipher Suite）被 mask 后，模型必须从上下文中推断——"前面有 TLS 版本号，后面有扩展字段，被 mask 的部分最可能是什么？"这迫使模型学习协议数据的结构化知识。

Task 2: Same-origin BURST Prediction (SBP)

对应 BERT 的 NSP（Next Sentence Prediction），但针对流量特性进行了重新设计。目标是建模 BURST 之间的联系——理解哪些数据段属于同一次通信交互。

任务设定：
给定两个 sub-BURST（A 和 B），判断它们是否来自同一个完整的 BURST。

- 正例（50%）：sub-BURST A 和 B 确实来自同一个 BURST 的前后半段
- 负例（50%）：sub-BURST B 来自随机不同的 BURST

取 [CLS] token 的最终隐藏向量，通过一个线性二分类器输出概率。
L_SBP = -[y·log(ŷ) + (1-y)·log(1-ŷ)]

总损失函数：

L_total = L_MBM + L_SBP

2.6 下游微调

ET-BERT 支持两种粒度的微调：

  • Packet-level (ET-BERT(packet))：以单个数据包作为输入，适合包级别的分类任务
  • Flow-level (ET-BERT(flow))：取 M 个连续数据包的拼接数据报作为输入（M=5），适合流级别的分类任务。将 M 个包的拼接数据报视为一个 BURST 输入模型。

取 [CLS] token 的输出表示，送入一个线性多分类器。微调时 所有参数解冻（这是一个关键的设计选择，后面会讨论其问题）。

3. 关键设计决策与原理

3.1 为什么不只用单字节（1-gram）？

单字节词表仅 256 个 token，虽然不会有 OOV 问题，但每个 token 承载的信息极少——一个 1500 字节的包会产生 1500 个 token，远超 BERT 的 512 token 限制。Bigram 将序列压缩一半（750 token），Tri-gram 压缩到 1/3（500 token），但 Tri-gram 的词表高达 16M 且极其稀疏。Bigram 的 65K 词表 + BPE 子词合并是序列长度和词表密度的最佳平衡。

3.2 为什么要去除 IP 头、端口等信息？

ET-BERT 明确去除了以太网头、IP 头、TCP/UDP 端口。设计意图是：这些信息不是加密流量的"内容"特征，而是"标识"特征——IP 地址和端口与具体主机/应用绑定，保留它们会让模型学到虚假相关性（如"所有来自 443 端口的流量都是 HTTPS"），而非真正的加密载荷模式。去除这些信息迫使模型从加密载荷本身学习区分性特征。

但实际效果存疑：后续的 Sugar 论文指出，即使去除了端口信息，仍有其他泄露渠道（TLS 握手明文字段如 SNI、包长模式、方向序列），这些非加密特征足以让模型在不去观察加密载荷的情况下达到高准确率。

3.3 为什么用双任务（MBM + SBP）而非单任务？

MBM 聚焦 BURST 内部的字节级依赖，SBP 聚焦 BURST 之间的交互关系。两者互补：MBM 学习"一个 ClientHello 长什么样"，SBP 学习"ClientHello 和 ServerHello 属于同一次握手"。消融实验证明，去掉 SBP 后性能显著下降，表明 BURST 间关系建模对下游任务（尤其是流级别分类）至关重要。

3.4 BURST vs 单包 vs 全流

这是最底层的数据粒度选择。

粒度优点缺点
单包简单直接，天然匹配包级分类ACK 等空包无信息；单包上下文不足以判断应用类型
全流包含完整交互上下文长度不可控，大量流超过 512 token；包含跨多轮交互的噪声
BURST自然语义边界，长度适中依赖方向切换作为边界信号，在单向流量中退化为全流

4.3 少样本学习 (Few-shot Learning)

这是证明预训练有效性的关键实验：

核心发现：当仅使用 10% 训练数据时，Deeppacket 的 F1 从 87.4% 暴跌至 47.2%（下降 40.2%），而 ET-BERT 仅从 97.4% 降至 91.6%。这说明预训练确实学到了可迁移的通用表示，大幅降低了下游任务的标注依赖。

4.4 跨平台泛化 (Cross-Platform Generalization)

一个值得关注的实验：在 Cross-Platform 数据集上，用 iOS 平台训练、测试 Android 平台（以及反向）。跨平台场景比同平台下降约 15-20%，但仍远超随机基线。这说明 ET-BERT 学到的表示具有一定的协议级别泛化能力（同一应用的 iOS 和 Android 版本共享底层协议交互模式），但平台特定的实现差异（如不同的 TLS 库、不同的包长度分布）仍然限制泛化。

5. 批判性分析

5.1 数据泄露问题（来自 Sugar / SoK 的致命批评）

问题一：Per-packet Split 导致严重数据泄露

ET-BERT 采用 per-packet（按包）而非 per-flow（按流）的 train/val/test 切分。这意味着同一条 TCP 流的多个数据包可能同时出现在训练集和测试集中。

为什么致命？同一条流的连续包共享大量信息——相同的 IP、端口、TLS 会话密钥上下文的包长模式。模型不需要学习"加密流量分类"，只需要学习"这个包和训练集里见过的那条流的包很像"即可取得高分。

数字证据：修复为 per-flow split 后，ET-BERT 的 F1 从 97.4% 崩溃至约 10.9%——这已经不是"下降"，而是"彻底失效"。
问题二：解冻微调掩盖了预训练的无用性

ET-BERT 在下游微调时解冻所有参数（而非仅微调分类头）。这就意味着即使预训练的表示完全无用，微调阶段的全参数更新也可以让模型从零学习。

数字证据：用随机初始化（无预训练权重）的 BERT 直接微调，性能几乎不变——97.1% vs 97.4%。这说明预训练带来的增益微乎其微，微调阶段的监督学习才是性能的主要来源。
问题三：非加密特征泄露

尽管去除了 IP 头和端口，但仍有大量非加密特征可供模型利用：TLS 握手明文部分的 SNI 字段、包长度分布和序列模式、方向切换模式、到达间隔等。这些特征与"加密载荷内容"无关，但足以区分不同的应用类型。

Sugar 实验证明：仅使用包长度序列（完全不看载荷），在相同数据集上就能达到接近 ET-BERT 的准确率。这意味着 ET-BERT 声称的"从加密载荷学习表示"可能只是烟雾弹——模型实际在使用包长等浅层特征。

5.2 密码学随机性分析（ET-BERT 的辩护尝试）

ET-BERT 在论文中尝试了密码学分析作为辩护：对 AES-GCM、AES-CBC、ChaCha20、RC4、3DES 五种加密算法的密文进行 15 种 NIST 随机性测试。结论是：这些算法在实践中并未达到完美的理论随机性——尤其是 RC4 和 3DES 数据集上分类性能更好。

但 Sugar 有力地反驳了这一点：TLS 1.3 强制使用 AEAD 密码（AES-GCM 或 ChaCha20-Poly1305），每个记录使用独立 IV（Initialization Vector），密文在理论上不应该包含任何可学习的模式。即使弱密码确实有非随机模式，那些模式也存在于已被淘汰的协议版本中——ET-BERT 在 TLS 1.3 数据集上 97.4% 的 F1 无法用密码学非随机性来解释，更可能来自于包长和方向等非加密特征。

5.3 其他局限

  • BURST 粒度依赖方向切换：在高度单向的流量（如大文件下载）中，BURST 可能退化为接近全流，导致 token 超长截断。
  • Bigram 编码缺乏语义：将两个相邻字节直接拼接忽视了字节的协议语义边界。一个 TLS 记录的结构为 ContentType(1B) + Version(2B) + Length(2B) + Payload，Bigram 可能正好切在字段中间——跨字段的 Bigram 组合没有实际协议含义。
  • 跨协议泛化未验证：预训练数据包含多种协议，但未系统验证模型从 TLS 学到的东西能否迁移到 QUIC 或其他协议。
  • 预训练数据与下游数据重叠：ISCX-VPN 同时出现在预训练和下游数据中。即使按包分流切分，同源数据的风格相似性也可能带来虚假的性能增益。

6. 历史定位与影响

6.1 历史贡献

ET-BERT 是第一个将 BERT 预训练+微调范式系统引入加密流量分类领域的工作。在此之前，该领域的深度学习方法（DeepPacket、FS-Net、DF等）都是纯监督学习。ET-BERT 证明了：

  1. 网络流量的字节序列可以像自然语言一样被 tokenize 并通过 Transformer 建模
  2. 自监督预训练在流量数据上是可行的（即使后来的分析揭示其增益有限）
  3. 预训练模型在少样本场景下相比纯监督方法有明显优势（这个结论即使经过更严格的评估也成立）

6.3 被 Sugar 批判后的学术价值再评估

Sugar（IEEE S&P 2025）和 SoK（2025）两篇论文对 ET-BERT 进行了严厉的批判，指出了数据泄露和预训练无效的问题。这些批评在方法论层面是成立的，但这并不意味着 ET-BERT 没有价值：

合理的定位：
- ET-BERT 的架构思路（BURST + Bigram + BPE + BERT pre-training）提供了流量预训练的基本框架
- ET-BERT 在预训练模型提出时公开了 CSTNET-TLS 1.3 数据集，为后续研究提供了宝贵基准
- ET-BERT 的失效模式（per-packet split、解冻微调、非加密特征泄露）本身成为宝贵教训，指导了后续如 TrafficFormer 等工作的实验设计
- ET-BERT 的少样本实验结论（预训练减少标注依赖）在采用更严格 split 的后继实验中也被验证

简言之，ET-BERT 的历史角色类似于"开路者"——它提出了正确的范式，但实验验证不够严谨。后续工作（TrafficFormer、YaTC 等）继承了其架构框架，但采用了更严格的评估协议。
