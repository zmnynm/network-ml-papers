TransSHAP / BERT meets Shapley: 将 Shapley 值解释适配到 Transformer 模型

打破 SHAP 对表格数据的依赖，首次将 Shapley 值框架系统性地移植到 BERT 类预训练语言模型的 token 级解释
Kokalj, Skrlj, Lavrac, Pollak & Robnik-Sikonja Jozef Stefan Institute, Slovenia EACL 2021 Hackashop

1. 动机：为什么 SHAP 不能直接用于 BERT？

SHAP (Lundberg & Lee, 2017) 在表格数据上取得了巨大成功，但其设计隐含了三个假设，这三个假设在 Transformer 模型上全部失效：

问题 1：特征不可独立操作

在表格数据中，"移除特征年龄"意味着将年龄列的值设为缺失值或均值——模型仍然可以接受这个残缺的输入。但在 BERT 中，你不能简单地"删除"一个 token。输入是一个 512 长度的序列，每个位置必须有且仅有一个 token。如果直接删除位置 i 的 token，序列变短了，位置编码错位了，模型的行为无法预测。

问题 2：零值掩码对 Transformer 无意义

原始 SHAP 的默认策略是将缺失特征设置为 0（或均值）。但在 Transformer 中，将 token 嵌入向量置零会产生一个BERT 从未在预训练中见过的输入——这个"零向量 token"不是任何真实词汇的表示，模型对其的响应是未定义的（out-of-distribution）。模型的输出变化可能来自"输入异常"而非"特征移除"，归因结果不可信。

问题 3：Token 之间存在深度上下文依赖

表格数据中，特征 age=35 的值独立于 income=50000——你可以独立地变动它们。但在 BERT 中，self-attention 机制使得每个 token 的表示依赖于序列中所有其他 token。这意味着即使你能技术上"替换"某个 token，其影响会通过 attention 传播到所有位置。原始 SHAP 的边际贡献框架假设特征可以独立扰动，这在强依赖的序列数据中是一个严重近似。

核心挑战：TransSHAP 需要回答的问题是——如何在保持 (a) Shapley 值的数学严格性、(b) Transformer 输入的有效性、(c) token 间上下文依赖 三者平衡的前提下，计算每个 token 对预测的公平贡献？

2. 核心适配：三项关键设计决策

适配 1：用 [MASK] 替换代替零值掩码

这是 TransSHAP 最关键的洞察。BERT 在预训练期间通过 Masked Language Modeling (MLM) 任务大量接触了 [MASK] token：15% 的输入 token 被替换为 [MASK]，模型学习从上下文预测原始词汇。这意味着：

  • [MASK] 是 BERT "熟悉"的输入——它不是 OOD 信号，而是预训练语境的自然组成部分
  • 将一个 token 替换为 [MASK] 相当于告诉模型"这个位置的信息暂时不可用，请从上下文推断"
  • 这完美对应了 Shapley 值中"特征缺失"的语义：不是删除了特征，而是用先验知识（上下文）填补了空缺

数学形式化：对于输入序列 x = (x_1, x_2, ..., x_n)，简化特征向量 z' ∈ {0,1}^n，定义映射函数 h_x：

h_x(z')_i = x_i     if z'_i = 1   (token 存在)
h_x(z')_i = [MASK]  if z'_i = 0   (token 缺失)

这与原始 SHAP 的零值/均值填充有本质区别：[MASK] 不是零向量，而是 BERT 词汇表中具有语义功能的特殊 token，其嵌入向量在预训练中被优化过。

适配 2：Kernel SHAP 采样策略

TransSHAP 采用 Kernel SHAP 的加权采样框架，但对采样进行了文本适配：

  • 特征定义：每个 WordPiece/BPE token = 一个特征。对于输入 "the cat sat on the mat"（7 个 token），特征空间维度 M=7
  • 子集采样：按 SHAP 核权重 π_x'(z') 采样简化特征向量 z'（哪些 token 保留，哪些替换为 [MASK]）
  • 模型评估：对每个采样的 z'，构造对应的掩码输入序列，前向传播 BERT 得到预测，然后求解加权线性回归得到 Shapley 值
  • 采样数量：通常使用 K=1000~5000 次模型评估（远小于 2^n，但对长文本仍是一个显著成本）

适配 3：Token 粒度的特征定义

TransSHAP 选择在 WordPiece token 级别进行解释——每个子词单元是一个独立的"特征"。这带来了粒度选择问题：

  • 优点：细粒度，可以捕捉子词级别的语义贡献（如 "un-" 前缀的否定含义）
  • 缺点：对用户不友好。例如 "embeddings" 被分割为 "em", "##bed", "##ding" 三个 token，分别解释后难以理解整体语义
  • 后续改进方向：可以向上聚合到词级别（sum 或平均 WordPiece Shapley 值），但聚合会丢失子词内部的贡献分布信息
设计权衡：[MASK] 替换策略是 TransSHAP 对 SHAP 框架最巧妙的本土化改造。它利用 BERT 预训练中内建的 MLM 能力，将"缺失值"问题转化为"mask 预测"问题——这正是 BERT 被训练要做的事。这一设计决策使得 Shapley 值的"缺失性公理"在 Transformer 语境中获得了自然的实现方式。

3. TransSHAP 可视化器：顺序呈现的革命

TransSHAP 的第二大贡献是提出了一种专门适配文本数据的 顺序可视化方案，这与传统 XAI 工具的做法截然不同。

3.1 传统方法的缺陷：排序列表

LIME 和 SHAP 的默认可视化将特征按重要性从高到低排序展示。这对表格数据是合理的——"年龄贡献 0.12，收入贡献 0.08，邮编贡献 0.02"的列表清晰有用。但对于文本：

  • 将 "The movie was surprisingly great despite the long runtime" 的 token 按 Shapley 值排序后，变成 "surprisingly" > "great" > "long" > "despite" > "the" > "movie" > ...
  • 这彻底破坏了语序信息——你不知道 "surprisingly" 修饰的是 "great" 还是 "long"
  • 文本中的上下文关系和句法结构在排序中完全丢失

3.2 TransSHAP 的顺序可视化

TransSHAP Visualizer 的核心原则：保持 token 的自然阅读顺序，用颜色编码重要性：

The
+0.01 → movie
+0.03 → was
+0.02 → surprisingly
+0.18 → great
+0.22 → despite
+0.01 → the
+0.01 → long
-0.09 → runtime
-0.11

设计中每个 token 的颜色由其 Shapley 值决定：

  • 绿色 = 正向贡献（推动预测朝向当前类别）
  • 红色 = 负向贡献（拉低预测，朝向对立类别）
  • 颜色饱和度/深浅 = 贡献的绝对值大小

3.3 顺序可视化的优势

人类读者可以同时看到两维信息：

  1. 空间/句法维度（从左到右的阅读顺序）：哪些位置的 token 重要？修饰词和中心词的关系是否在解释中得到体现？
  2. 重要性维度（颜色深浅）：哪些 token 贡献最大？正负贡献如何分布？

作者通过用户实验证实：参与者使用顺序可视化识别关键证据 token 的准确率显著高于使用排序列表（p < 0.01）。

4. 完整示例：情感分析中的 TransSHAP

输入句子："This movie was surprisingly great despite the long runtime."
模型预测：Positive（积极情感），置信度 0.87
问题：模型为什么判定这句话为积极？哪些词驱动了这个预测？

4.1 TransSHAP 计算流程（逐步演示）

Step 1: Tokenization（WordPiece）

["[CLS]", "this", "movie", "was", "surprising", "##ly", "great", "despite", "the", "long", "runtime", ".", "[SEP]"]
→ M = 13 个 token（包括特殊 token [CLS] 和 [SEP]）

Step 2: 按 SHAP 核采样简化特征向量

假设采样 K=1000 个 z' 向量。每个 z' 是一个 13 位二进制向量。例如：

z' = [1, 1, 1, 0, 1, 1, 1, 1, 1, 0, 0, 1, 1]
→ 保留 "[CLS]", "this", "movie", "surprising", "##ly", "great", "despite", "the", ".", "[SEP]"
→ 将 "was", "long", "runtime" 替换为 [MASK]

Step 3: 模型前向传播

masked_input = ["[CLS]", "this", "movie", "[MASK]", "surprising", "##ly", "great", "despite", "the", "[MASK]", "[MASK]", ".", "[SEP]"]
→ BERT forward → 预测输出 f(h_x(z')) = 0.62 (仍判为 positive，但置信度下降)

注意：移除 "long" 和 "runtime" 后预测从 0.87 下降到 0.62，说明这两个词对 negative 方向有重要影响——它们的存在抑制了 positive 预测。

Step 4: 加权线性回归求解 Shapley 值

用 1000 次评估的 (z', f(h_x(z'))) 数据拟合加权线性回归，得到每个 token 的 Shapley 值：

TokenShapley 值方向语义解释
"great"+0.22正向核心积极信号词
"surprising"+"##ly"+0.18正向强化积极程度（出乎意料地好）
"movie"+0.03正向中性主体词，轻微正向（电影语境）
"this"+0.01正向限定词，几乎无贡献
"was"+0.02正向系动词，弱正向关联
"despite"+0.01正向轻微正向（暗示转折被克服）
"the"+0.01正向冠词，几乎无贡献
"long"-0.09负向负面信号（长——通常不好）
"runtime"-0.11负向负面信号（片长过长）
"."+0.01正向句末标点，几乎无贡献

4.2 解读

TransSHAP 的解释清晰地揭示了模型的推理逻辑：模型识别到 "great" 和 "surprisingly" 是强烈的积极信号（正向贡献合计 +0.40），而 "long runtime" 是负面信号（负向贡献合计 -0.20）。尽管存在负面信号，正向信号的强度压倒性地主导了预测。

4.3 TransSHAP vs 传统可视化对比

传统排序列表：
  1. great: +0.22
  2. surprisingly: +0.18
  3. runtime: -0.11
  4. long: -0.09
  5. movie: +0.03
  6. was: +0.02
  7. ... (其余)
→ 丢失了 "surprisingly" 修饰 "great"、 "long" 修饰 "runtime" 的句法关系
TransSHAP 顺序可视化：
this movie was surprisingly great despite the long runtime .
→ 保留了语序和上下文关系，读者可以直观定位重要短语

6. 局限性与未解决问题

6.1 计算成本

Kernel SHAP 的核心瓶颈：每个采样点 z' 需要一次完整的 BERT 前向传播。对于一个 100-token 的输入，通常需要 K=2000~5000 次评估来达到可接受的信噪比。每次 BERT forward pass 在 GPU 上约 10~50 毫秒，总计需要 20~250 秒 来解释一个样本。这在交互式应用中不可接受，在批量离线分析中也需要大量 GPU 资源。

6.2 WordPiece 粒度的用户体验问题

人类理解的是"单词"而非"子词片段"。TransSHAP 对 "surprisingly"（在 WordPiece 中分为 "surprising" + "##ly"）给出两个独立的 Shapley 值，但用户想知道的是"surprisingly 这个词的总体贡献"。简单的相加聚合（+0.07 + 0.11 = +0.18）可能合理，但没有理论保证这种聚合的合法性。

6.3 Token 间相关性的忽略

与原始 SHAP 一样，TransSHAP 在计算边际贡献时假设 token 可以独立地被 [MASK] 替换。但在实践中，n-gram 短语（如 "New York"、"long runtime"）中的 token 高度共现。同时 Mask 掉 "New" 而保留 "York" 可能产生一个 BERT 不认识的奇怪输入，使得 Shapley 值偏离真实语义。

6.4 长文本的采样效率

当文本长度增加到 200~500 tokens（BERT 的最大输入 512），采样空间 2^M 变成天文数字（2^200 ≈ 10^60）。Kernel SHAP 的采样效率在这体量上严重退化——几千次采样在 10^60 的空间中几乎为零覆盖。对于长文档分析（如安全日志、网络流量序列），这构成了根本性限制。

6.5 解释的因果性问题

Shapley 值本身是相关性量度而非因果性量度。即使 TransSHAP 正确计算了每个 token 对预测的贡献（在合作博弈的意义上），这也不意味着改变该 token 就会导致预测变化（在因果意义上）。对于安全关键应用（如 NIDS 解释），这种关联性-因果性的混淆可能导致错误的响应决策。

待解决的核心矛盾：Transformer 的优势在于 self-attention 捕捉的 token 间丰富交互——但这恰恰是 Shapley 值框架最不适应的部分。Shapley 值假设贡献可以加性分解，而 Transformer 的力量来自非线性的、全对全的交互建模。TransSHAP 通过 [MASK] 策略优雅地绕过了这一问题，但本质矛盾并未消除：我们用一个加性框架（Shapley）去解释一个非加性系统（Transformer），信息损失是必然的。

7. 对流量预训练模型的启示

TransSHAP 的设计原则可以直接迁移到网络安全领域的流量 Transformer 模型（如 ET-BERT）。两者的相似性体现在：

维度NLP BERTTraffic BERT (ET-BERT)TransSHAP 适配策略
基本单元 WordPiece token BURST token（流量突发段） 将每个 BURST token 视为一个"特征"，进行 [MASK] 替换
掩码语义 [MASK] 在 MLM 预训练中优化 ET-BERT 使用类似的 Masked Traffic Modeling [MASK] 策略可以直接继承——ET-BERT 也被训练以从上下文预测被掩码的流量段
序列长度 通常 50~200 tokens 通常 32~128 BURST tokens（流量包数量有限） 较短的序列降低了计算成本，Kernel SHAP 采样效率更高
解释目标 哪些词驱动了情感/分类 哪些流量段驱动了攻击/正常判断 识别关键恶意流量特征（如异常包长序列、特殊端口等）
顺序可视化 保持自然阅读顺序 保持包到达的时间顺序 时间顺序可视化——安全分析师可以看到攻击发生在哪个时间窗口
实际应用前景：将 TransSHAP 应用于 ET-BERT，可以使安全分析师看到一段网络流量中哪些 BURST token（即哪些时间段的流量模式）最能解释模型的"恶意/正常"判定。这对于生成取证证据、理解误报原因、以及生成对抗性流量的防御策略都具有直接价值。此外，类似于 TransSHAP 的顺序可视化，对于流量数据可以设计时间轴可视化——横轴按包到达顺序排列，颜色深度表示每个 BURST token 对攻击分类的贡献量。

TransSHAP 对流量模型的局限性也同样存在：流量 token 之间的时间依赖性（一个包的语义取决于前后包）比文本 token 之间的句法依赖更强，这使得 Shapley 值的"独立扰动"假设在流量数据中近似度更大。但即便如此，[MASK] 替换策略仍然比零值掩码或随机替换合理得多——因为它利用了预训练模型的 mask 推理能力来填补缺失的流量信息。
